🟦 1. Kyverno (Kubernetes-native approach)

“Starting with Kyverno.

Kyverno is a Kubernetes-native policy engine, meaning everything is defined as standard Kubernetes resources.

Key points:

Policies are written in YAML, so there’s no new language like Rego to learn
It supports not just validation, but also:
Mutation (auto-add defaults)
Generation (create resources like NetworkPolicies)
Image verification
It fits very naturally into a GitOps model

So from an architect’s perspective, Kyverno is ideal when you want developer-friendly, Kubernetes-centric policy-as-code.”

👉 You can add:
“In real projects, this reduces friction because platform and app teams already understand YAML.”

🟩 2. Azure Policy Add-on (Cloud governance approach)

“Next is the Azure Policy add-on for AKS.

This is a Microsoft-managed solution, built on top of OPA Gatekeeper, but abstracted for Azure users.

Key characteristics:

Policies are managed centrally via Azure (Portal, CLI, Terraform)
Tight integration with:
Azure RBAC
Policy Initiatives
Defender for Cloud
Uses a two-layer model:
ConstraintTemplate (logic)
Constraint (parameters)

From an architecture standpoint, this is best suited for:

Enterprise governance
Multi-subscription standardization
Teams already aligned with Azure-native controls

👉 You can add:
“It’s less flexible than native tools but very strong for compliance and audit.”

🟨 3. ValidationAdmissionPolicy (Native Kubernetes control plane)

“Finally, ValidationAdmissionPolicy.

This is the most native approach, built directly into the Kubernetes API server.

Key points:

Uses CEL (Common Expression Language) for defining rules
No additional components—no webhooks, no extra pods
Very low operational overhead
But currently focused on validation only
(no mutation or generation like Kyverno)

From an architect’s perspective:

It’s ideal for lightweight, built-in guardrails
But not sufficient alone for complex policy use cases

👉 You can add:
“It’s powerful, but still evolving compared to Kyverno or OPA-based solutions.”

🔄 Strong transition to next discussion

“So if you look at these three:

Kyverno → Kubernetes-native, flexible, developer-friendly
Azure Policy → centralized governance, enterprise control
ValidationAdmissionPolicy → lightweight, built-in enforcement

The real challenge is not which one is ‘best’, but:
👉 which one fits your platform architecture and operating model

And that’s what we’ll compare next.”
